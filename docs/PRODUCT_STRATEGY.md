## 1. 市场与竞品分析

### 1.1 ACG 工具/社区平台市场概览

2026 年的中文 ACG 互联网生态中，用户的核心需求集中在四个维度：**发现内容、追踪进度、讨论交流、获取资源**。市场上存在多个定位不同的产品，但没有任何一款产品实现了这四个维度的无缝整合。

从用户行为链条来看，一个典型的深度 ACG 用户的使用路径如下：

```
发现新番 → 决定追哪部 → 标记进度 → 找资源观看 → 评分/写评论 → 与同好讨论
   │            │            │           │              │             │
B站/AniList  Bangumi/     Bangumi/  Hikarinagi/    Bangumi/      巴哈姆特/
 当季新番表   AniList      AniList   Mikan/DMHY    B站评论区      Bangumi小组
```

用户被迫在多个平台之间跳转，每个平台只能满足链路中的 1-2 个环节。ANISpace 的机会在于：**成为这条链路的"中枢"**，不是替代任何一个平台，而是在它们之上提供统一的体验层。

### 1.2 主要竞品深度对比

#### 1.2.1 Bangumi（bgm.tv）—— 最核心竞品

**优势**：
- **数据库深度**：10 年+ 用户贡献的数据，拥有中文 ACG 圈最完整的人物、角色、制作组关系数据。条目关联关系（前传/续集/外传/相同世界观）比任何其他平台都丰富。
- **评分公信力**：Bangumi 的 10 分制评分在中文 ACG 圈有较高的参考价值，尤其是动画分类下的排名常被引用。
- **社区沉淀**：小组系统和维基贡献者社区活跃，长尾条目（冷门 Galgame、老番）的讨论质量高。
- **API 可用性**：提供 `/v0` 系列 RESTful API（虽然非官方公开但长期稳定），是技术集成的可行入口。

**劣势**：
- **UI/UX 陈旧**：界面设计停留在 2010 年代初期风格，对新用户不友好。移动端体验尤其差。
- **推荐系统缺失**：依赖"喜欢这部的人也喜欢"的简单协同过滤，无个性化推荐能力，冷门作品很难被发现。
- **平台锁定**：用户标记数据一旦积累到数百条，迁移成本极高——这既是 Bangumi 的护城河，也是用户被"绑架"的不满来源。
- **功能固化**：五态标记体系（想看/在看/看过/搁置/抛弃）十年未变，不支持自定义标签或分类。
- **API 隐忧**：`/v0` 接口并非官方公开 API，没有 SLA 承诺，理论上有随时调整的风险。

**ANISpace 的应对**：
- 不做替代，做"聚合增强层"——用户不需要放弃 Bangumi 的标记数据，ANISpace 通过 OAuth 同步后提供更好的浏览、搜索、推荐体验。
- 同时实施"渐进独立"策略——用户的每一次标记通过 `auto_enrich` 机制沉淀到自有 `bangumi_subjects` 表，逐步降低对外部 API 的依赖。
- 在 Bangumi 做不到的维度上发力：AI 推荐、自然语言搜索、自定义收藏体系。

#### 1.2.2 Bilibili（bilibili.com）—— 内容生态巨头

**优势**：
- **内容护城河**：独家版权番剧 + UGC 内容生态（动画杂谈、MAD、手书、翻唱等），是中文 ACG 圈最大的内容消费入口。
- **弹幕文化**：弹幕创造了独特的"共同观看"体验，社交属性强。
- **推荐算法**：基于海量用户行为数据的推荐系统成熟，但更多服务于平台利益（推广自制内容和广告），而非用户最佳体验。
- **用户基数**：MAU 超过 3 亿，网络效应无与伦比。

**劣势**：
- **版权删减**：大量引进番剧存在画面删减或内容修改，核心用户对此极度不满。
- **社区泛化**：随着用户泛化，弹幕质量和社区氛围下降，老用户流失。
- **不专注**：B 站已从 ACG 垂直平台演变为综合视频平台，ACG 核心用户体验被稀释。
- **追番功能薄弱**：追番功能简陋，不支持评分，不支持跨平台数据。

**ANISpace 的应对**：
- 不与 B 站在内容层面竞争——不做视频托管，不做推荐流。
- 利用"跨平台资源雷达"功能帮用户找到番剧在哪个平台能看、哪里有未删减版本，解决 B 站用户"找完整版"的刚需。
- 做 B 站忽略的"深度管理"——B 站的追番功能只是一个简单的列表，ANISpace 提供的是 Notion 级别的收藏管理。

#### 1.2.3 AniList（anilist.co）—— 国际化标杆

**优势**：
- **UI/UX 设计出色**：界面现代、美观，统计图表和可视化非常精美。特别是用户个人页面的数据报告设计感强。
- **GraphQL API 开放**：提供功能强大的 GraphQL API，开发者生态活跃，第三方客户端丰富。
- **评分灵活性**：支持 1-100 分制（可切换为星级/笑脸等），自定义列表功能灵活。
- **当季新番表**：新番浏览体验优于 Bangumi，信息展示直观，适合快速决策。

**劣势**：
- **中文覆盖差**：条目的中文名、中文简介覆盖率低，标签和分类体系以英文为主，不适合中文母语用户。
- **社区在中国无基础**：社交功能（Global Feed、用户关注）在中国几乎没有用户网络效应。
- **视频资源缺失**：不提供任何视频可用性信息，纯粹的追踪/记录工具。

**ANISpace 的应对**：
- 学习 AniList 的数据可视化设计（用于后续的"个人数据洞察报告"功能）。
- 在"统一追番仪表盘"功能中接入 AniList 作为可选数据源，帮助同时在 AniList 标记的用户统一管理。
- 以中文条目数据（Bangumi 主干 + 自有数据库）为核心优势，提供 AniList 无法比拟的中文体验。

#### 1.2.4 巴哈姆特（gamer.com.tw）—— 社区文化壁垒

**优势**：
- **论坛文化深厚**：20 年+ 的 ACG 综合讨论区，用户粘性极高，讨论氛围（哈拉区）是中文圈最有特色的论坛文化之一。
- **动画疯**：旗下的正版动画串流平台（巴哈姆特动画疯）在台湾地区有较大用户基数，免费+1080P 模式受欢迎。
- **游戏数据库**：游戏条目收录和讨论在中文圈最强，ACG 数据库与论坛深度绑定。

**劣势**：
- **地域限制**：核心用户集中在台湾，大陆用户访问有网络障碍（部分需翻墙）和文化差异。
- **界面古老**：论坛式 UI 不符合现代用户的浏览习惯，移动端体验差。
- **追番管理薄弱**：收藏夹功能简陋，无评分体系，不支持数据导出或跨平台。

**ANISpace 的应对**：
- 不竞争论坛文化（这不是技术能弥补的 20 年积累），保持现有"放課後"论坛作为轻量社交即可。
- 将巴哈姆特动画疯作为"跨平台资源雷达"的重要数据源之一，帮助大陆用户发现台湾地区的可观看资源。
- 在游戏条目的术语和 UI 上借鉴巴哈姆特的本地化表达（如"玩过"而非"看过"）。

#### 1.2.5 Hikarinagi（hikarinagi.com）—— 垂直工具型

**优势**：
- **视频聚合专注**：专注做一件事——匹配番剧的视频源（字幕组资源），搜索效率和准确度不错。
- **轻量快速**：无社区功能，纯粹的工具型产品，加载速度快。
- **字幕组信息**：提供字幕组名称、格式、来源等详细元数据。

**劣势**：
- **功能单一**：只有视频源搜索，没有条目管理、评分、社区等任何其他功能，用户用完即走。
- **数据覆盖有限**：主要覆盖当季和热门番剧，冷门/老番覆盖率不高。

**ANISpace 的应对**：
- 视频源搜索作为 ANISpace 的功能之一而非全部，已有完善的 `SourceManager` 和 `MediaMatchList` 体系可复用。
- "跨平台资源雷达"在 Hikarinagi 的基础上新增"正版平台可用性"维度，覆盖 Hikarinagi 不做的正版渠道。
- 用户在 ANISpace 完成"找番 → 标记 → 找资源 → 观看"的完整闭环，不需要像用 Hikarinagi 那样来回切换平台。

#### 1.2.6 鲲Galgame论坛（kungalv.com）—— 垂类社区

**优势**：
- **Galgame 深度**：Galgame 领域的讨论质量、资源丰富度远超通用 ACG 平台。
- **评测文化**：长文评测、感想文类型的 UGC 内容质量高。
- **用户信任**：在 Galgame 玩家群体中有较高的社区信任度。

**劣势**：
- **品类限制**：仅覆盖 Galgame，用户规模天花板明显。
- **现代功能缺乏**：无 API、无数据导出、无个性化推荐。
- **界面陈旧**：传统论坛式 UI，移动端体验差。

**ANISpace 的应对**：
- Musashi 创作者平台可以向 Galgame 方向做定向优化（已有 Galgame 下载管理、章节管理等组件）。
- 但不过度依赖 Galgame 赛道——ANISpace 的定位是全品类 ACG 个人管理工具。
- 在条目详情页对 Galgame 类型做专门的字段适配（如"游戏时长"、"路线数量"、"攻略链接"等）。

**关键发现**：个人工具化（自定义管理 + AI 驱动 + 跨平台聚合）是当前所有竞品都未充分覆盖的空白地带。

### 1.4 用户痛点总结

基于竞品分析和目标用户行为研究，识别出四个核心痛点：

| 痛点 | 现状 | 严重程度 | 被解决的竞品 |
|------|------|----------|-------------|
| **多平台数据割裂** | 用户在 Bangumi、AniList、B 站等地分散标记，进度和评分不互通 | ★★★★★ | 无 |
| **消费管理粗放** | 五态标记体系过于简单，无法满足多维度管理需求（如按声优、按制作公司、按补番优先级分类） | ★★★★☆ | 无 |
| **找资源困难** | "某部番在哪能看"需要去多个平台手动搜索，信息分散且不实时 | ★★★★☆ | 部分（Hikarinagi 解决字幕组搜索） |
| **缺少个人数据洞察** | 用户无法从自己的消费数据中获得有价值的分析（品味画像、年度趋势、好友重合度） | ★★★☆☆ | 部分（AniList 有基础统计图） |

## 2. ANISpace 现状评估

### 2.1 现有架构与功能矩阵

ANISpace 目前已实现 16 个功能模块，技术架构为 React 19 + Vite 前端部署在 GitHub Pages，Cloudflare Worker + D1 作为后端，形成了完整的前后端体系。

**现有功能模块清单**：

| 模块 | 功能 | 状态 | 用户覆盖 |
|------|------|------|----------|
| M-01 认证 | Bangumi/GitHub OAuth 登录，JWT 会话管理 | 已上线 | 全量 |
| M-02 资料 | 用户资料编辑、头像上传、隐私设置 | 已上线 | 全量 |
| M-03 论坛 | 分板块发帖/回复/点赞（放課後） | 已上线 | 社交用户 |
| M-04 世界频道 | 全站实时消息广场（世界线） | 已上线 | 社交用户 |
| M-05 邮箱 | 用户间异步通信（D-Mail） | 已上线 | 社交用户 |
| M-06 好友空间 | 好友动态发布/可见（LeMU） | 已上线 | 社交用户 |
| M-07 通知 | 系统通知聚合 | 已上线 | 社交用户 |
| M-08 Wiki | 条目搜索、排行榜、随机推荐（禁書目錄） | 已上线 | 全量 |
| M-09 视频 | 多源视频搜索聚合 | 已上线 | 全量 |
| M-10 音乐 | 音乐搜索/播放/歌单 | 已上线 | 全量 |
| M-11 资讯 | ACG 新闻聚合（毒电波！！） | 已上线 | 全量 |
| M-12 TouchGal | Galgame 论坛嵌入浏览 | 已上线 | 全量 |
| M-13 社团 | 兴趣群组创建与管理（Tea Time!） | 已上线 | 社交用户 |
| M-14 留言板 | 社区留言墙 | 已上线 | 全量 |
| M-15 Amadeus | AI 角色对话（Navi） | 已上线 | 全量 |
| M-16 追番/收藏 | 条目标记与评分 | 已上线 | 全量 |

**关键基础设施（已就绪）**：

| 组件 | 用途 | 对本次战略的复用价值 |
|------|------|---------------------|
| `bangumi_subjects` 表 | 条目完整数据本地存储 | "渐进独立"策略的核心数据基础 |
| `auto_enrich` 机制 | 用户标记时自动拉取 Bangumi API 补充完整数据 | 用户每次标记都在帮 ANISpace 充实数据库 |
| `bangumi_index` 表 | 5.5 万+ 条目的轻量索引 | 搜索功能的本地数据源 |
| GLM-4-Flash API 接入 | AI 聊天能力 | 可直接用于 AI 推荐和自然语言搜索 |
| Bangumi OAuth | 用户授权登录 | 可扩展为多平台数据同步的入口 |
| Worker Cron 系统 | 定时任务（bangumi-data 同步、资讯爬取） | 可用于平台可用性数据的定时采集 |
| IDB 缓存体系 | IndexedDB LRU 缓存 (200条/30min) | 可用于推荐列表和搜索结果的客户端缓存 |
| 视频源系统 | SourceManager/MatchEngine/多源适配器 | 可直接用于字幕组资源搜索 |
| `CollectionMarkService` | 收藏标记管理（含本地备份） | 自定义收藏体系的数据写入层 |

### 2.2 现有短板与改进空间

**vs 本次战略方向，当前缺失的核心能力**：

1. **无跨平台数据聚合**：虽然接入了 Bangumi OAuth，但仅用于登录，不同步用户已有的 Bangumi 标记数据
2. **无推荐系统**：条目发现依赖搜索和排行榜，没有个性化推荐能力
3. **收藏管理固定**：只有五态 + 评分的固定模式，不支持自定义字段或视图
4. **搜索维度单一**：只支持关键词搜索，无法按情节描述、多条件组合搜索
5. **资源信息孤立**：视频源搜索独立运行，没有与条目详情页的"观看途径"信息整合

**优势（可直接复用的基础）**：

1. **桌面 OS 式 UI**：WindowManager + Dock 架构天然适合"多面板工作台"式的仪表盘设计
2. **数据已开始积累**：`auto_enrich` 机制意味着每个用户在标记时都在为平台数据库做贡献
3. **Cloudflare 基础设施**：Worker + D1 的 serverless 架构，开发和运维成本低，扩展灵活
4. **已有视频多源系统**：Mikan/DMHY/MacCMS 等源的适配器可以直接用于资源雷达
5. **GLM API 已集成**：不需要从零接入 AI 能力

## 3. 产品愿景与战略定位

### 3.1 一句话定位

> **ANISpace = 你的 ACG 个人数据中心 + AI 副驾驶**
>
> 跨平台聚合你的所有追番、收藏、评分数据，用 AI 帮你发现下一个心头好，像 Notion 一样自由组织你的二次元世界。

### 3.2 产品战略框架

```
┌──────────────────────────────────────────────────────────┐
│                       ANISpace                             │
│                                                            │
│  ┌──────────────┐  ┌──────────────┐  ┌─────────────────┐  │
│  │  数据聚合层    │  │  AI 引擎     │  │  个人工具箱      │  │
│  │              │  │              │  │                 │  │
│  │ ● Bangumi    │  │ ● 智能推荐    │  │ ● 自定义分类     │  │
```

### 3.3 "渐进独立"数据策略

**核心原则**：用户永远不需要"迁移"数据。ANISpace 作为聚合增强层，自动同步外部平台数据；同时用户的每一次交互都在为自有数据库贡献内容。

**阶段一（当前→v1.1）**：以 Bangumi API 为主要数据源，通过 OAuth 同步用户标记数据。同时 `auto_enrich` 机制持续将条目完整数据写入自有 `bangumi_subjects` 表。AniList 作为辅助数据源接入。

**阶段二（v1.2→v2.0）**：自有数据库条目数达到 1 万+，覆盖 95% 的主流条目。对外部 API 的依赖降低到补充长尾条目和新番。开始向用户提供"数据自主权"价值——用户的标记数据存储在 ANISpace，可以导出到任何平台。

**阶段三（v2.0+）**：自有数据库足够丰富，可以完全独立运行。社区纠错和完善机制成熟。ANISpace 成为中文 ACG 圈的新兴数据库基础设施，同时保持对其他平台的聚合能力。

```
              AI 原生

             /    \
            /      \
           / ANISpace\

    数据聚合          个人工具化
```

## 4. 五大创新功能详细设计

### 4.1 统一追番仪表盘

#### 4.1.1 功能概述

统一追番仪表盘是 ANISpace v1.0 的首页级功能。它解决的是 ACG 用户"多平台数据割裂"的核心痛点——用户可能在 Bangumi 有 300 条标记、在 AniList 有 50 条、在 B 站有追番列表，这些数据互不相通。

仪表盘将这些跨平台数据聚合到一个视图中，展示统一的追番进度、状态统计和观看计划，让用户首次拥有 ACG 消费的"全局视图"。

#### 4.1.2 用户场景

**场景一：跨平台进度同步**

用户小明在 Bangumi 标记了 300+ 条动画，在 AniList 有 50 条（主要是当季新番，因为 AniList 的当季新番表更直观），偶尔在 B 站看番时也会顺手点"追番"。某天他想继续补《86-不存在的战区-》，但不记得看到第几集了。

- **现状**：打开 Bangumi 查看"在看"列表，没找到（因为在 AniList 标记的）。打开 AniList 找到后，又需要切换回 Bangumi 才能看评分和评论。
- **ANISpace**：打开仪表盘，看到跨平台聚合的"在看"列表。《86》显示进度：第 17 集/共 23 集，来源 AniList，3 天前更新。旁边的"观看途径"面板直接显示 B 站和字幕组资源链接。一站式完成"查进度 → 继续看"。

**场景二：新番季决策**

每季新番开始，用户需要在 50+ 部新番中决定追哪几部。现有方式需要：在 AniList 看新番表 → 在 Bangumi 看评分预测 → 在 B 站看 PV → 来回切换。

- **ANISpace**：仪表盘的"新番雷达"面板自动聚合当季新番，标注每部的 Bangumi 评分预测、AniList 关注人数、以及基于用户偏好的 AI 匹配度百分比。用户可以在一个页面内完成所有信息收集 → 快速标记"想看"。

**场景三：补番计划管理**

用户积压了 200+ 部"想看"，但没有有效的优先级管理工具。

- **ANISpace**：仪表盘的"补番队列"面板按用户自定义的优先级排序（如"年份 + 评分 + 集数"），直观展示"下一部该补什么"。支持拖拽排序。

#### 4.1.3 技术设计

**数据流架构**：

```
用户授权 OAuth
    │
    ├── Bangumi OAuth ──→ Worker 拉取 /v0/users/{username}/collections
    │                       └─→ 按 subject_id 存入，标记 source='bangumi'
    │
    ├── AniList OAuth ──→ Worker 拉取 GraphQL MediaListCollection
    │                       └─→ 通过 AniList ID → Bangumi ID 映射表去重
    │                       └─→ 存入，标记 source='anilist'
    │
    └── 本地标记 ────────→ 直接写入 D1 collections 表
                            └─→ source='anispace'
```

**核心数据模型（新增表）**：

```sql
-- 外部平台账户关联表
CREATE TABLE user_platform_accounts (
  id INTEGER PRIMARY KEY AUTOINCREMENT,
  user_id INTEGER NOT NULL,
  platform TEXT NOT NULL,          -- 'bangumi', 'anilist', 'mal'
  platform_user_id TEXT NOT NULL,
  platform_username TEXT,
  access_token TEXT,               -- 加密存储
  refresh_token TEXT,
  token_expires_at TEXT,
  last_synced_at TEXT,
  sync_status TEXT DEFAULT 'idle', -- 'idle', 'syncing', 'error'
  sync_error TEXT,
  created_at TEXT DEFAULT (datetime('now')),
  UNIQUE(user_id, platform)
);

-- 跨平台收藏标记聚合表
-- 注意：不新建表，而是修改现有 collections 表，新增 source 字段
ALTER TABLE collections ADD COLUMN source TEXT DEFAULT 'anispace';
ALTER TABLE collections ADD COLUMN source_id TEXT;        -- 源平台上的记录ID
ALTER TABLE collections ADD COLUMN external_data TEXT;     -- 源平台的额外数据JSON
```

**去重策略（核心难点）**：

以 Bangumi Subject ID 为**主键**进行跨平台条目映射：

1. **AniList → Bangumi 映射**：利用 `bangumi_subjects` 表中已有的 `sites` 字段（Bangumi API 返回的跨站链接），建立 `anilist_id ↔ bangumi_id` 映射。对于无映射的条目，使用 `(title_cn, type, year)` 三元组做模糊匹配。
2. **MAL → Bangumi 映射**：利用 Jikan API 的 `external_links` 字段获取 Bangumi ID 映射。
3. **冲突解决**：同一 subject_id 在多个平台有不同状态/评分时，以"最近更新时间"为准，但在 `external_data` 字段保留所有平台的历史记录。

**同步策略**：

| 触发时机 | 行为 | 限流措施 |
|----------|------|----------|
| 用户授权平台账号后（首次） | 全量拉取用户该平台的所有标记（可能数百条），分批请求 | Bangumi: 50条/页，1req/s；AniList: GraphQL 单次查询量限制 |
| 每次打开仪表盘 | 增量同步：只拉取 `updated_at > last_synced_at` 的条目 | 同上 |
| Worker Cron（每 6 小时） | 后台对已授权用户执行增量同步 | 每次 Cron 只处理前 50 个待同步用户 |
| 用户手动点击"刷新" | 强制全量重新同步 | 限制 5 分钟内只能触发一次 |

**前端数据查询接口**：

```
GET /api/dashboard/summary
  → 返回: { total, watching, want_to_watch, watched, on_hold, dropped }
  → 数据来源: 三个平台的 UNION 结果按 subject_id 去重后统计

GET /api/dashboard/collections?status=watching&sort=recent&page=1&limit=20
  → 返回: 去重合并后的标记列表，每条包含 source 标签和平台数据
```

### 4.2 AI 智能推荐引擎

#### 4.2.1 功能概述

AI 智能推荐引擎是 ANISpace 在 AI 原生方向上的核心能力。与竞品的"喜欢这部的人也喜欢"简单推荐不同，ANISpace 的推荐引擎通过三层漏斗（协同过滤 + 内容相似度 + AI 语义理解），为用户提供三种类型的推荐：

- **热门相似**（40%）：基于协同过滤的稳健推荐，确保基本质量
- **冷门发掘**（40%）：评分人数少但口碑好的隐藏佳作，这是核心差异化
- **跨类型惊喜**（20%）：刻意推荐用户习惯类型之外的高分作品，增加发现感

每条推荐附带自然语言的"推荐理由"，而非简单的"相似度分数"。

#### 4.2.2 用户场景

**场景一：看完好番后的"下一部看什么"**

小红刚看完《孤独摇滚》，被"社恐主角在音乐中找到归属"的故事深深打动。她想知道还有什么类似的作品。但她已经在 Bangumi 看过了所有热门推荐（《轻音少女》、《K-ON!》等），需要的是**小众但神似**的作品。

- **Bangumi 现状**：推荐的都是大热门，大部分她都看过了。
- **ANISpace**：AI 推荐引擎分析她的评分模式，发现她高分作品的共同特征不是"音乐"标签，而是"孤独者在群体中找到自我认同"的叙事主题。推荐了《空之音》（评分人数仅 287，但口碑 8.0+），推荐理由："军武外壳下的日常温情，氛围与《摇曳露营》异曲同工"。小红给了 8 分，评价"差点错过"。

**场景二：新用户冷启动**

小刚刚加入 ANISpace，没有任何收藏历史，无法进行个性化推荐。

- **ANISpace**：引导他完成"口味测试"——展示 10 部有代表性的作品（《钢之炼金术师》、《Clannad》、《凉宫春日的忧郁》、《EVA》、《魔法少女小圆》等），他快速表态：喜欢/无感/不喜欢。基于这 10 个回答，AI 立即生成第一版推荐列表。后续随着他标记更多条目，推荐的个性化程度逐步提高。

**场景三：新番季"我要追什么"**

每季有 50+ 部新番播出，用户需要快速决策。传统方式是一一查看 PV、简介、Staff 阵容。

- **ANISpace**：AI 引擎对每部新番自动计算"匹配度百分比"，基于用户历史偏好。匹配度 > 80% 的标为"强烈推荐"；50-80% 标为"可能喜欢"；< 30% 标为"可能不感兴趣"但不隐藏（用户可以自己判断）。推荐不只看热门作品——冷门但匹配度高的新番也会被推到前面。

#### 4.2.3 技术设计

**推荐架构：三层漏斗模型**

```

│  → 这是稳健推荐，保证基本质量           │
│                                      │

               │ 候选池: ~50 条
               ▼

│                                      │

│  → 在 bangumi_subjects 中匹配相似标签   │
```

**AI Prompt 设计（核心）**：

```
System Prompt:
你是一个资深 ACG 文化分析专家，擅长通过用户的收藏历史
分析其深层品味偏好。

任务：根据以下信息，为用户生成个性化推荐列表。

【用户高分条目（评分 ≥ 8）】
- 《孤独摇滚》(9分): 社恐少女加入乐队，在音乐中找到归属的喜剧
- 《三月的狮子》(10分): 将棋棋手在孤独中被他人温暖的治愈故事
- 《比宇宙更远的地方》(9分): 少女们前往南极完成青春冒险
- 《摇曳露营》(8分): 少女露营的悠闲温馨日常
-《少女终末旅行》(9分): 末世背景下两个少女的旅途与哲学思考

【候选推荐池（每个条目含名称、简介、标签）】
{候选条目 JSON 数据，约 50-100 条}

【输出要求】
1. 偏好分析（50字内）：用一句话概括用户的深层偏好模式
2. 推荐 15 条，分为三类:
   A. 热门相似 (6条): 与用户高分条目相似的热门作品
   B. 冷门发掘 (6条): 评分人数 < 500 但口碑好的隐藏精品
   C. 跨类型惊喜 (3条): 不在用户习惯类型内但可能打动他的作品
3. 每条包含:
   - subject_id（如果候选池中有）
   - 匹配度%（85-99%之间）
   - 一句话推荐理由（25字内，不要说"类似XXX"，而要说明共鸣点）
4. 严格按 JSON 格式输出
```

**冷启动策略**：

当用户无标记数据时（`collections_count < 5`），触发"口味测试"流程：

1. 预选 10 部"探测作品"：涵盖不同年代（90s/00s/10s/20s）、不同类型（热血/治愈/悬疑/日常/科幻）、不同知名度（热门/冷门）
2. 用户对每部表态：👍 喜欢 / 😐 无感 / 👎 不喜欢
3. 将结果序列化为偏好向量：`{类型偏好, 年代偏好, 主题偏好, 节奏偏好}`
4. AI 基于此向量生成首批推荐

后续每增加 5 条用户标记，自动重新计算推荐列表并降低口味测试结果权重（线性衰减至 20 条标记后完全不参考）。

**推荐缓存策略**：

| 缓存层 | 存储位置 | TTL | 触发刷新 |
|--------|----------|-----|----------|
| 用户推荐列表 | D1 `user_recommendations` 表 | 24h | 新增标记后异步重新计算 |
| AI 推荐理由 | 同上（text 字段） | 24h | 同上 |
| 前端展示缓存 | IndexedDB (IDB) | 1h | 用户主动点击"刷新"或过期 |

#### 4.2.5 实现难度评估

| 模块 | 难度 | 关键依赖 | 说明 |
|------|------|----------|------|
| 协同过滤（SQL实现） | ★★☆☆☆ | D1 SQL JOIN 性能 | 在万级用户数据量下 SQL 可行 |
| 标签相似度计算 | ★★☆☆☆ | bangumi_subjects.tags | 现有的 JSON 字段可直接使用 |
| AI 推荐 (GLM API) | ★★★☆☆ | GLM-4-Flash API | 核心是 prompt engineering，非模型训练 |
| 冷启动口味测试 | ★☆☆☆☆ | 纯前端交互 | 10 道题 + 前端评分组件 |
| 推荐缓存与更新 | ★★☆☆☆ | Worker Cron + D1 | 预计算而非实时推荐，降低 AI 调用成本 |
| **整体** | **★★★☆☆** | — | 预估 1.5-2 周 |

**关键风险**：
- **GLM API 调用成本**：每次生成推荐需传入 50-100 条候选条目的摘要（约 5000-10000 token），按 GLM-4-Flash 的定价（约 ¥0.001/1K token），每次推荐成本约 ¥0.05-0.1。以 1000 DAU、每人每天刷新 1 次计算，日均成本约 ¥50-100。需监控成本并设置单用户每日调用上限。
- **协同过滤冷启动**：新平台用户标记量少时，第一层协同过滤效果差。依赖第二层（内容相似度）和第三层（AI）补足。随着数据积累自动改善。
- **Prompt 稳定性**：AI 输出的 JSON 格式可能偶尔不稳定（格式错误、字段缺失），需要在 Worker 层做输出校验 + 解析失败时重试（最多 2 次）。

### 4.3 自定义收藏体系

#### 4.3.1 功能概述

自定义收藏体系是 ANISpace"个人工具化"战略的旗舰功能。它借鉴 Notion 数据库的设计哲学——将"数据属性"和"展示视图"分离，让用户可以：

- 创建自定义字段（标签、下拉选择、数字、日期、评分等类型）
- 构建多条件筛选器（AND/OR 组合）
- 在列表视图和看板视图之间切换
- 使用预设模板快速上手

这是与 Bangumi/AniList/MAL 最根本的差异化——所有竞品都是固定字段（状态 + 评分 + 标签），只有 ANISpace 让用户像使用 Notion 一样自由管理自己的 ACG 收藏。

#### 4.3.2 用户场景

**场景一：声优厨的多维管理**

小美是声优爱好者。她不仅记录"看过什么"，还想按声优筛选收藏。她的自定义字段包括：

- "主役声优"（标签类型，多选）
- "配角声优"（标签类型，多选）
- "声优表现评分"（评分类型，1-5）

她经常使用筛选器："显示所有早见沙织主役的、我还没看的动画"。

**场景二：季度补番计划**

每季结束时，用户想回顾"本季我追了哪些、弃了哪些、为什么弃"。传统平台只能按状态分类，无法记录"为什么弃"。

- 自定义字段"弃番原因"（下拉选择：画风不喜 / 剧情无聊 / 太虐不敢看 / 时间不够 / 其他）
- 筛选"本季 + 抛弃 + 画风不喜"，帮助用户总结自己的口味偏好

**场景三：补番优先级管理**

用户积压了 200 部"想看"，需要系统性地排优先级。自定义字段：

- "补番优先级"（下拉选择：高 / 中 / 低）
- "预计所需时间"（数字类型，单位：集数）
- "想看年份"（年份，记录是哪一年加入的"想看"）

筛选器："补番优先级 = 高 AND 类型 = 动画 AND 集数 ≤ 13" → 优先补短篇高分作品。

**筛选器查询构建**：

当用户应用筛选条件时，前端将条件发送到后端，后端动态构建 SQL：

```javascript
// Worker 端伪代码
function buildFilterQuery(userId, filters) {
  let sql = `
    SELECT c.*, s.name, s.name_cn, s.type, s.score, s.image
    FROM collections c
    LEFT JOIN bangumi_subjects s ON c.subject_id = s.id
  `;

  const conditions = [];
  const params = [];

  // 系统字段筛选（直接查 collections 表）
  filters.forEach(filter => {
    switch (filter.field) {
      case 'status':
        conditions.push(`c.status = ?`);
        params.push(filter.value);
        break;
      case 'rating_min':
        conditions.push(`c.rating >= ?`);
        params.push(filter.value);
        break;
      case 'type':
        conditions.push(`s.type = ?`);
        params.push(filter.value);
        break;
      // ...
    }
  });

  // 自定义字段筛选（JOIN user_custom_field_values）
  const customFilters = filters.filter(f => f.isCustom);
  if (customFilters.length > 0) {
    customFilters.forEach((filter, i) => {
      sql += ` JOIN user_custom_field_values cfv${i}
               ON c.subject_id = cfv${i}.subject_id
               AND cfv${i}.field_id = ?
               AND cfv${i}.user_id = ?`;
      params.push(filter.fieldId, userId);

      if (filter.fieldType === 'tag') {
        conditions.push(`cfv${i}.value LIKE ?`);
        params.push(`%${filter.value}%`);
      } else {
        conditions.push(`cfv${i}.value = ?`);
        params.push(filter.value);
      }
    });
  }

  if (conditions.length > 0) {
    sql += ` WHERE c.user_id = ? AND ` + conditions.join(' AND ');
    params.unshift(userId);
  } else {
    sql += ` WHERE c.user_id = ?`;
    params.push(userId);
  }

  return { sql, params };
}
```

- 这是与所有竞品最根本的差异化——Bangumi 做了 10 年+ 也未曾考虑让用户自定义管理维度
- 对于深度用户来说，自定义分类是"从能用变成好用"的关键一跃

- **前端复杂度**：自定义字段变更时，所有视图需要同步更新。建议使用统一的 `useCustomFields` hook + React Context 做状态管理。

- **看板视图拖拽**：可以选用 react-beautiful-dnd（已停止维护但稳定）或 framer-motion 内置的 drag 手势（项目已引入）。

### 4.4 AI 自然语言搜索

#### 4.4.1 功能概述

AI 自然语言搜索让用户用日常语言描述自己想找的作品，而非使用关键词或筛选器。它解决了两个传统搜索无法处理的场景：

1. **模糊记忆搜索**：用户记不清名字，只记得情节片段或设定元素
2. **多维条件组合**：用户想同时按类型、年代、评分、标签、情节等多个维度搜索，传统方式需要逐个设置筛选器

技术路线上，该功能通过 GLM-4-Flash 将自然语言转换为结构化查询 JSON，然后在本地数据库执行精确搜索，最后对结果进行 AI 重排序和说明生成。

#### 4.4.2 用户场景

**场景一：童年回忆搜索**

用户依稀记得小时候在电视上看过一部动画："主角到了一个全是动物的王国，好像是被什么力量带过去的，他要在那里生存"。他只记得这些碎片，完全想不起名字。

- **传统搜索**：输入"动物 王国 生存"在 Bangumi 搜索，结果有几百条，逐一翻看。
- **ANISpace**：输入"主角到了一个全是动物的王国，要在那里生存的动画"→ AI 解析为语义搜索 → 在条目简介中匹配 → 返回《兽王星》，匹配度 92%，附说明："根据描述，这部作品设定在一个野兽统治的星球，人类是下级生物，符合'动物王国'和'生存'主题。"

**场景二：情节定位**

用户想看某一集："哪部动画有一集是主角在学园祭上弹吉他的？前面几集很压抑，这一集突然变轻松了"。

- **ANISpace**：AI 解析为语义查询 → 在条目简介、章节数据、用户评论中搜索 → 返回《三月的狮子》第二季第 4 集（校园祭篇），匹配度 85%。

**场景三：复杂条件查询**

用户想看"2010年以前的、评分大于 8.5 的、不是战斗题材的、每集20分钟左右、最好是讲日常生活的动画"。

- **传统搜索**：需要在 Bangumi 手动设置年份、类型、排序，再逐个点进去看评分和简介。
- **ANISpace**：一键输入 → AI 解析为：`{ year_max: 2010, score_min: 8.5, tags_exclude: ["战斗"], type: 2, eps_avg: 24 }` → 返回 8 条精确匹配结果。

**AI 解析 Prompt 设计**：

```
System Prompt:
你是一个 ACG 搜索解析器。将用户的自然语言搜索请求
转换为结构化的 JSON 查询。只输出 JSON，不要有任何其他文字。

支持的查询字段：
- keyword: 标题/简介关键词搜索（字符串数组）
- type: 条目类型 (1=书籍, 2=动画, 3=音乐, 4=游戏, 6=三次元)
- score_min: 最低评分 (0-10)
- score_max: 最高评分 (0-10)

  "query_type": "structured" 或 "semantic",
  "structured": {

  },
  "semantic_hint": "如果无法结构化，用50字概括用户在找什么"
}

规则：
- 只填充你能确认的字段，不要猜测
- 如果用户主要描述情节/设定而非具体筛选条件，使用 semantic 类型
- tags_include/tags_exclude 的值应该是标准中文 ACG 标签
- 年份范围从用户描述中推断，如"老番"→ year_max=2010
```

**结构化查询执行**：

```javascript
// Worker 端伪代码
async function executeStructuredSearch(structured) {
  let sql = `SELECT * FROM bangumi_subjects WHERE 1=1`;
  const params = [];

  if (structured.type) {
    sql += ` AND type = ?`;
    params.push(structured.type);
  }
  if (structured.score_min) {
    sql += ` AND score >= ?`;
    params.push(structured.score_min);
  }
  if (structured.year_max) {
    sql += ` AND CAST(json_extract(infobox, '$.放送开始') AS TEXT) <= ?`;
    params.push(`${structured.year_max}-12-31`);
  }
  if (structured.tags_exclude?.length) {
    structured.tags_exclude.forEach(tag => {
      sql += ` AND tags NOT LIKE ?`;
      params.push(`%${tag}%`);
    });
  }
  // ... 更多条件

  sql += ` ORDER BY score DESC LIMIT 20`;
  return await db.prepare(sql).bind(...params).all();
}
```

**降级方案：模糊匹配 + AI 重排序**：

当结构化查询命中数 < 3 条时：

1. 对 `bangumi_subjects` 的 `name`, `name_cn`, `summary` 字段做 `LIKE '%关键词%'` 搜索
2. 将命中的条目（最多 30 条）的 `summary` 和 `tags` 发送给 AI
3. AI 根据 `semantic_hint` 对候选列表进行重排序
4. 返回排序后的 top 10

**D1 全文搜索限制与解决方案**：

D1（SQLite 兼容）不原生支持 FTS（全文搜索引擎）。搜索优化方案：

1. **v1.0 方案（LIKE 查询）**：在 `name`, `name_cn` 字段上使用 `LIKE '%keyword%'` 进行子串匹配。在 5 万条数据量下，带索引的 LIKE 查询性能可接受（< 200ms）。
2. **v1.1 优化方案（Worker 内存索引）**：通过 Worker Cron 定期从 `bangumi_subjects` 重建一个内存搜索索引（`Map<string, Set<number>>`），在 Worker 的全局作用域中维护，实现 O(1) 的关键词 → subject_ids 查找。
3. **v2.0 方案（外部搜索服务）**：如果数据量达到 10 万+，考虑引入 MeiliSearch 或 Algolia 做专业的全文搜索。

### 4.5 跨平台资源雷达

#### 4.5.1 功能概述

跨平台资源雷达是 ANISpace 在"找资源困难"痛点上的针对性解决方案。它在条目详情页展示该作品在各平台的可用性信息，涵盖两个维度：

- **正版平台**：Bilibili、巴哈姆特动画疯、Crunchyroll、Netflix 等平台的番剧可用性（是否在线、是否有删减、是否需要会员、区域限制）
- **字幕组资源**：通过 Mikan/DMHY 等渠道获取的字幕组发布信息（字幕组名、格式、集数、种子/磁力链接）

与现有视频源系统（`SourceManager`/`MediaMatchList`）的关系：资源雷达在现有系统基础上新增"正版平台"维度的信息整合，而非替代现有功能。

#### 4.5.2 用户场景

**场景一：找正版观看渠道**

用户想看《命运石之门》，打开 B 站发现已下架，在百度云找了一圈没有高清资源，最后花了 20 分钟在巴哈姆特动画疯找到能看的（但需要台湾 IP）。

- **ANISpace**：打开《命运石之门》条目详情页，资源雷达面板显示：
  - Bilibili：已下架（曾上线过，共 24 集）
  - 巴哈姆特动画疯：可观看 · 全 24 集 · 免费 · 需要台湾 IP
  - Netflix：仅日本区可用 · 日语配音 · 无中文字幕
  - 字幕组资源：喵萌奶茶屋 BD 1080P 合集 23.5GB

一站式完成渠道对比，用户不再需要挨个平台搜索。

**场景二：追新番决定"在哪看"**

每季新番，用户需要在 B 站、巴哈姆特、甚至自己找字幕组之间选择。

- **ANISpace**：新番条目页的资源雷达面板直接显示：
  - Bilibili：已引进 · 每周日 01:00 更新 · 有删减（第 3 集画面调整）
  - 巴哈姆特动画疯：已引进 · 每周日 00:30 更新 · 无删减 · 免费
  - 字幕组（已有 3 个组在跟）：喵萌奶茶屋 / 动漫国 / 悠哈璃羽

用户一目了然：要无删减 → 巴哈姆特 / 字幕组；要方便 → B 站。

**场景三：字幕组版本选择**

某部老番有多个字幕组的版本，用户不确定哪个翻译质量更好。

- **ANISpace**：展示各字幕组的发布时间、格式（TV/BD）、文件大小，并可点击展开评论区看其他用户的评价（如"喵萌奶茶屋的翻译更准确，建议选这个"）。用户点击即可复制磁力链接或下载种子文件。

**实时字幕组搜索**：

字幕组资源不预缓存（因为信息实时变化，且体积小），而是在用户查看条目详情时实时搜索：

```javascript
// Worker 端伪代码
async function getFansubResources(subjectId, subjectName) {
  // 尝试从缓存表获取（1小时内的结果）
  const cached = await db.prepare(
    `SELECT * FROM subject_fansub_resources
     WHERE subject_id = ?
     AND created_at > datetime('now', '-1 hour')
     ORDER BY publish_date DESC`
  ).bind(subjectId).all();

  if (cached.results?.length > 0) {
    return cached.results;
  }

  // 实时搜索各源
  const results = await Promise.allSettled([
    searchMikan(subjectName),      // 已有 SourceManager
    searchDMHY(subjectName),       // 已有 SourceManager
  ]);

  // 存储搜索结果
  const allResults = results
    .filter(r => r.status === 'fulfilled')
    .flatMap(r => r.value);

  // 批量写入缓存表
  const stmt = db.prepare(
    `INSERT OR REPLACE INTO subject_fansub_resources
     (subject_id, fansub_group, episode, title, magnet_link, torrent_url, file_size, format, resolution, publish_date, source)
     VALUES (?, ?, ?, ?, ?, ?, ?, ?, ?, ?, ?)`
  );
  const batch = allResults.map(r => stmt.bind(
    subjectId, r.group, r.episode, r.title, r.magnet, r.torrent,
    r.size, r.format, r.resolution, r.date, r.source
  ));
  await db.batch(batch);

  return allResults;
}
```

### 6.1 技术可行性总结

| 功能 | 核心依赖 | 依赖成熟度 | 风险评估 |
|------|----------|-----------|----------|
| 统一追番仪表盘 | Bangumi OAuth + AniList OAuth + D1 | 成熟 | 🟡 中（跨平台去重映射） |
| AI 智能推荐 | GLM-4-Flash API + D1 查询 | 成熟 | 🟢 低（纯 Prompt Engineering） |
| 自定义收藏体系 | D1 + React 状态管理 | 成熟 | 🟡 中（前端复杂度） |
| AI 自然语言搜索 | GLM-4-Flash + D1 LIKE | 成熟 | 🟡 中（D1 全文搜索限制） |
| 跨平台资源雷达 | 各平台公开接口 + HTML 解析 | 部分成熟 | 🟡 中（HTML 解析脆弱） |

### 6.2 关键风险与缓解措施

#### 风险一：Bangumi API 依赖风险（影响：高 · 概率：低）

**风险描述**：ANISpace 的条目数据和用户标记数据高度依赖 Bangumi `/v0` 系列 API。虽然该 API 已稳定运行数年，但 Bangumi 团队未有官方 SLA 承诺。如果 Bangumi 突然限制 API 访问或调整接口，ANISpace 的核心功能（搜索、条目详情、数据同步）将受影响。

**缓解措施**：
1. "渐进独立"策略是根本解决方案——通过 `auto_enrich` 机制持续将 Bangumi 数据沉淀到自有数据库
2. 保持多数据源能力——AniList API 作为备用条目数据源（虽然中文覆盖率低，但可以兜底）
3. 自有 `bangumi_index` 表（5.5 万+ 条）已经是 Bangumi 关闭 API 时的可用降级方案
4. 与 Bangumi 团队建立联系，了解 API 路线图（长期）

#### 风险二：跨平台条目去重准确率（影响：中 · 概率：高）

**风险描述**：AniList、MAL 等平台的条目 ID 与 Bangumi Subject ID 无法 100% 完美映射。部分条目在各个平台的名称、类型分类、年份信息不一致（如 OVA/剧场版/TV 版分类差异、中文译名不统一等），导致自动化去重映射可能出现错误。

**缓解措施**：
1. 建立多层映射策略：`sites` 字段精确匹配 → 名称+类型+年份三元组模糊匹配 → 手动关联
2. 提供"手动关联"功能：用户可以自己将两个平台上的同一部作品关联起来，关联数据反哺给系统
3. 数据隔离：不同平台的标记数据分开存储（`source` 字段标记来源），去重在查询层完成，不修改原始数据
4. 接受不完美的映射率（目标 95%），而非追求 100%

#### 风险三：D1 数据库性能瓶颈（影响：中 · 概率：中）

**风险描述**：D1 虽然基于 SQLite，但作为 Cloudflare 的分布式数据库服务，其查询性能受到 Workers 的 CPU 时间限制（10ms 墙时间，实际可更长但有约束）。自定义收藏体系的多表 JOIN + 动态筛选可能导致复杂查询超时。中文 LIKE 全文搜索性能可能不足。

**缓解措施**：
1. 对 `user_custom_field_values` 建 `(user_id, field_id, value)` 联合索引
2. 筛选查询优化：先查主表（collections），再批量查自定义字段值（用 `IN` 子句），而非多层 JOIN
3. v1.1 实现 Worker 内存搜索索引，替代 LIKE 查询
4. 分页设置默认 `limit=20`，避免单次查询返回过多数据
5. 如果 D1 性能确实成为瓶颈，考虑引入 Cloudflare KV 做缓存层（已有 IndexedDB 做客户端缓存）

#### 风险四：AI 调用成本（影响：低 · 概率：中）

**风险描述**：AI 推荐和自然语言搜索依赖 GLM-4-Flash API，每次调用需要消耗 token。随着用户量增长，AI 调用成本可能超出预期。

**缓解措施**：
1. 推荐列表预计算而非实时生成：用户新增标记后异步更新推荐列表，缓存 24 小时
2. 设置单用户每日 AI 调用上限：推荐刷新 3 次/天，搜索 20 次/天
3. 三层推荐漏斗中，前两层（协同过滤 + 标签相似度）不消耗 AI token，只有第三层需要
4. 成本预估：1000 DAU × 每日 1 次推荐刷新 × ¥0.05/次 = ¥50/天 = ¥1500/月（可控）
5. 如成本确实过高，可引入本地推理模型（WebLLM）作为降级方案（v2.0 考虑）

#### 风险五：正版平台数据采集的脆弱性（影响：中 · 概率：中）

**风险描述**：巴哈姆特动画疯等平台的数据采集依赖 HTML 页面解析，页面结构随时可能变化导致采集失败。部分平台（如 Bilibili 国际版）的接口可能有反爬机制。

**缓解措施**：
1. Worker 日志中记录每次采集的成功/失败状态
2. 采集连续失败 3 次 → 触发告警 → 人工检查并修复解析规则
3. 采集失败的数据不影响其他平台的展示——每个平台独立采集、独立展示
4. 字幕组资源走实时搜索（不走缓存），不需要定期采集
5. 长期关注各平台是否开放官方 API（如巴哈姆特计划中的 Open API）

### 8.2 开发流程建议

```
需求评审 ──→ 技术设计 ──→ 开发 ──→ 自测 ──→ Code Review ──→ 提测 ──→ 上线
   │                                                        │
   └──────────── 产品经理参与评审 + 验收 ───────────────────────┘
```

**关键实践**：

1. **每周同步会**（30 分钟）：周一早上，回顾上周进度、本周计划、识别阻塞
2. **功能验收标准**：每个功能开工前写好验收清单（如仪表盘：跨平台同步成功率 ≥ 95%）
3. **渐进式交付**：每个功能先交付 MVP 版本上线，然后迭代优化——而非等到完美才发布
4. **用户反馈闭环**：v1.0 发布后两周内，主动联系早期用户做 1v1 访谈，收集反馈用于 v1.1 规划
